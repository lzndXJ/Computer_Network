## 3.1 Data Link Layer Services / 数据链路层服务

The data link layer provides node-to-node (hop-to-hop) delivery. It uses the physical layer's raw bit service to create a reliable link.

> **中文**：数据链路层提供节点到节点（逐跳）的交付服务。它利用物理层的原始比特服务来创建一条可靠的链路。

| Service Type / 服务类型 | Description / 描述 | Example / 示例 |
|-------------|-------------|---------|
| **Unacknowledged connectionless** / **无确认无连接** | No connection, no ACK; corrupted frames discarded silently / 无连接，无确认；损坏的帧被静默丢弃 | **Ethernet** / **以太网** |
| **Acknowledged connectionless** / **有确认无连接** | No connection, but each frame is acknowledged individually / 无连接，但每一帧都被单独确认 | **Wi-Fi (802.11)** |
| **Acknowledged connection-oriented** / **有确认面向连接** | Connection established, numbered frames exchanged, acknowledged / 建立连接，交换带序号的帧并进行确认 | **Used in some WAN links** / **用于某些广域网链路** |

## 3.2 Framing: Four Methods / 成帧：四种方法

The data link layer must break the bit stream into discrete **frames**【帧】.

> **中文**：数据链路层必须将比特流分割成离散的**帧**。

### Method 1: Byte Count / 方法1：字节计数法

- First field of the frame indicates the **number of bytes** in the frame / 帧的第一个字段指示该帧的**字节数**
- **Fatal flaw**: If the count byte is corrupted, all subsequent frames are misaligned (resynchronization is nearly impossible) / **致命缺陷**：如果计数字节出错，后续所有帧都会错位（几乎无法重新同步）
- **No longer used** / **已不再使用**

> **中文**：字节计数法在帧的首部设置一个计数字段，标明该帧包含多少字节。其致命问题在于，一旦计数字节在传输中出错，收端将无法正确划分后续所有帧的边界，且几乎不可能重新同步。因此这种方法已不再使用。

### Method 2: Flag Bytes with Byte Stuffing (Character Stuffing) / 方法2：标志字节与字节填充（字符填充）

- Use special flag byte (e.g., `0x7E`) to mark frame start/end / 使用特殊的标志字节（如`0x7E`）标记帧的开始和结束
- If flag byte appears in data, precede it with an **escape byte** (`ESC`) / 如果标志字节出现在数据中，在其前面添加一个**转义字节**（`ESC`）
- If `ESC` appears in data, precede it with another `ESC` / 如果`ESC`出现在数据中，在其前面再添加一个`ESC`
- Used by **PPP** protocol / 用于**PPP**协议

> **中文**：字节填充法使用特殊标志字节（如`0x7E`）标示帧的起止。若数据中出现了与标志字节相同的值，发送端在其前面插入一个转义字节`ESC`；若数据中出现了`ESC`本身，则在其前面再插入一个`ESC`。接收端收到后删除插入的转义字节，恢复原始数据。PPP协议使用此方法。

### Method 3: Flag Bits with Bit Stuffing ★★★ (Must Master / 必须掌握)

**Used by HDLC protocol** / **用于HDLC协议**。The flag pattern is: **`01111110`** (six consecutive 1s) / 标志模式为：**`01111110`**（6个连续的1）。

**Transparency rule** / **透明传输规则**：To prevent the data from accidentally containing the flag / 为了防止数据中意外出现标志模式：
- **Sender** / **发送端**：After every **5 consecutive 1 bits** in the data, **automatically insert a 0** (bit stuffing) / 在数据中每遇到**连续5个1**，就**自动插入一个0**（比特填充）
- **Receiver** / **接收端**：After receiving **5 consecutive 1 bits**, **automatically delete the following 0** (bit destuffing) / 收到**连续5个1**后，**自动删除后面的0**（比特删除）

**Example** / **示例** — Sender wants to transmit / 发送端要发送：`0110111111110111110`

1. Scan for 5 consecutive 1s and note where to insert 0s / 扫描连续5个1，标记需要插入0的位置
2. Original / 原始数据：`0110 1111111 10 11111 0`
3. After bit stuffing (0 inserted after each group of 5 consecutive 1s) / 比特填充后（在每组连续5个1后插入0）：
   `0110 11111 0 11 0 10 11111 0 0`
4. Frame with flags / 加上标志后的帧：`01111110` + stuffed_data + `01111110`

> **中文**：HDLC协议使用比特填充法。标志模式为`01111110`（即6个连续1）。发送端在数据中每检测到连续5个1，就自动插入一个0；接收端收到连续5个1后，自动删除后面的0。这样数据中就不会出现类似标志模式`01111110`的序列，实现了透明传输。

**Key exam point** / **考试要点**：Bit stuffing is used when the **physical layer has no special encoding** to mark frame boundaries. The bit-stuffing approach is **easy to implement in hardware**.

**In practice** / **实际应用**:
- Framing methods are often **combined** to achieve both transparency and efficiency / 成帧方法常常**结合使用**以达到透明传输和效率的平衡
- **USB** uses a combination of bit stuffing and other techniques / **USB**协议就使用了比特填充与其他技术的结合

> **中文**：在实际系统中，成帧方法经常组合使用。例如USB协议就结合了比特填充等多种技术来实现帧边界标识。

> **中文**：当**物理层没有特殊编码**来标记帧边界时，使用比特填充。比特填充方法**易于用硬件实现**。

**What the receiver does** / **接收端的工作**：Given `01101111101101111100` / 给定数据`01101111101101111100`：
- Remove stuffed 0s after each group of 5 consecutive 1s / 删除每组连续5个1后面的填充0
- Recover / 恢复：`0110 1111111 10 11111 0 → 0110111111110111110`

### Method 4: Physical Layer Coding Violations / 方法4：物理层编码违例

- Use invalid physical encoding to mark frame boundaries / 使用非法的物理编码来标记帧边界
- e.g., in Manchester encoding, a transition always occurs in the middle of each bit; a miss can signal frame start/end / 例如，在曼彻斯特编码中，每个比特中间必然有电平跳变；没有跳变则可表示帧的起止
- No overhead for bit/byte stuffing, but depends on physical layer / 没有比特/字节填充的开销，但依赖于物理层

> **中文**：物理层编码违例方法利用物理层中非法的编码信号来标记帧边界。例如在曼彻斯特编码中，每个比特中间都有电平跳变，若检测到无跳变则代表帧的开始或结束。此方法没有填充开销，但依赖于物理层的能力。

## 3.3 差错检测与纠正

### 3.3.1 关键概念

- **码字**：在信道上传输的 $n$ 位比特单元，由 $m$ 位数据位和 $r$ 位校验位组成，因此 $n = m + r$。
  - **编码**：把 $m$ 位数据变成 $n$ 位码字，也就是加入 $r$ 位校验位。
  - **解码**：接收方根据 $n$ 位码字恢复数据，并进行差错检测或纠正。
- **海明距离/码距**：两个码字之间不同比特位的个数。
  - 计算方法：把两个码字逐位异或，结果中 `1` 的个数就是海明距离。
- **码率**：数据位占总码字长度的比例，公式为 $m/n$。
- **码的最小距离**：任意两个有效码字之间的最小海明距离，决定检错和纠错能力。
- **突发错误**：从第一个出错比特到最后一个出错比特之间的一段连续比特序列，中间的比特可能出错，也可能没有出错。
- **突发错误检测能力**：一个编码能够保证检测出的最大突发错误长度。
- **误比特率 BER**：单个比特在传输中发生翻转的概率。
- **误包率 PER**：一个 $N$ 位分组中至少有一位出错的概率。若误比特率为 $BER$，则：

$$
PER = 1 - (1 - BER)^N
$$

### 3.3.2 海明距离规则

要检测最多 $d$ 位错误，需要满足：

$$
\boxed{海明距离 \ge d + 1}
$$

要纠正最多 $d$ 位错误，需要满足：

$$
\boxed{海明距离 \ge 2d + 1}
$$

理解方式：

- 检错：错误后的码字不能刚好变成另一个合法码字。
- 纠错：错误后的码字必须离原来的合法码字最近，不能落入其他合法码字的纠错范围。

### 3.3.3 奇偶校验

**简单奇偶校验**：在数据后面加 1 位校验位，使整个码字中 `1` 的个数满足约定的奇偶性。

- **偶校验**：数据位和校验位中 `1` 的总数为偶数。
- **奇校验**：数据位和校验位中 `1` 的总数为奇数。

特点：

- 简单奇偶校验码的最小海明距离为 2。
- 可以检测单比特错误。
- 可以检测奇数个位错误。
- 不能检测偶数个位错误。
- 不能纠正错误，只能发现异常。

例子：

```text
数据：1011001
其中 1 的个数：4 个

偶校验：校验位 = 0，发送 10110010
奇校验：校验位 = 1，发送 10110011
```

**二维奇偶校验**：把数据排列成矩阵，对每一行和每一列分别加入校验位。

```text
数据按行排列：
  1 0 1 1 | 1   ← 行校验位
  0 1 0 1 | 0   ← 行校验位
  1 1 0 0 | 0   ← 行校验位
  --------+--
  0 0 1 0       ← 列校验位
```

二维奇偶校验的优点：

- 如果只有 1 位出错，可以通过“出错行”和“出错列”的交点定位错误位，并进行纠正。
- 能检测大多数多比特错误模式。
- 比简单奇偶校验更强，但需要更多校验位。

### 3.3.4 循环冗余校验 CRC ★★★

CRC 是最常用的检错码之一。它把比特串看成二进制多项式的系数，并在模 2 运算下做多项式除法。模 2 运算的核心就是异或：没有进位，也没有借位。

CRC 的基本思想：

1. 发送方和接收方事先约定一个生成多项式 $G(x)$，其次数为 $r$。
2. 发送方在数据后面补 $r$ 个 `0`。
3. 用补零后的数据除以 $G(x)$，采用模 2 除法。
4. 得到的 $r$ 位余数就是 CRC 校验和。
5. 发送方把“原始数据 + CRC 余数”一起发送。
6. 接收方用同一个 $G(x)$ 去除收到的比特串。
7. 如果余数为 0，认为没有检测到错误；如果余数不为 0，说明检测到错误。

公式写法：

- 原始数据多项式：$M(x)$
- 生成多项式：$G(x)$，次数为 $r$
- CRC 余数：$R(x)$
- 发送码字：

$$
T(x) = M(x) \cdot x^r + R(x)
$$

注意：CRC 只能检错，不能直接纠错。

常见生成多项式：

| 名称 | 生成多项式 | 校验位数 |
|---|---|---:|
| CRC-8 | $x^8 + x^2 + x + 1$ | 8 |
| CRC-12 | $x^{12} + x^{11} + x^3 + x^2 + x + 1$ | 12 |
| CRC-16 | $x^{16} + x^{15} + x^2 + 1$ | 16 |
| CRC-CCITT | $x^{16} + x^{12} + x^5 + 1$ | 16 |
| CRC-32 | $x^{32} + x^{26} + x^{23} + x^{22} + x^{16} + x^{12} + x^{11} + x^{10} + x^8 + x^7 + x^5 + x^4 + x^2 + x + 1$ | 32 |

CRC 计算例子：

```text
数据：10010101010
生成多项式对应比特串：10101
```

因为 `10101` 是 5 位，所以生成多项式次数 $r = 4$，先在数据后补 4 个 `0`：

```text
10010101010 0000
```

然后用模 2 除法除以 `10101`，得到余数：

```text
1110
```

所以发送帧为：

```text
10010101010 1110
```

接收方再用 `10101` 去除收到的完整比特串。如果余数为 0，则没有检测到错误；如果余数不为 0，则说明传输中出现了错误。

CRC 的硬件实现通常使用线性反馈移位寄存器和异或门，异或门的位置由生成多项式中的各项决定。

### CRC 的检错性质 ★

CRC 的检错能力取决于生成多项式 $G(x)$ 的选择。

**性质 1：可以检测所有单比特错误**

单比特错误可以写成：

$$
E(x) = x^j
$$

只要 $G(x)$ 至少有两项，$x^j$ 就不可能被 $G(x)$ 整除，因此余数不为 0，可以检测出来。

**性质 2：适当选择 $G(x)$ 时，可以检测所有双比特错误**

双比特错误可以写成：

$$
E(x) = x^i + x^j = x^j(x^{i-j}+1)
$$

如果生成多项式选择得好，使 $x^k + 1$ 在相关范围内不能被 $G(x)$ 整除，就可以检测双比特错误。

**性质 3：如果 $G(x)$ 含有 $(x+1)$ 因子，可以检测所有奇数个位错误**

原因是：奇数个位错误对应的错误多项式不能被 $(x+1)$ 整除，因此也不能被含有 $(x+1)$ 因子的 $G(x)$ 整除。

**性质 4：可以检测长度不超过 $r$ 的所有突发错误**

其中 $r$ 是生成多项式 $G(x)$ 的次数。也就是说：

$$
\boxed{CRC 可以检测所有长度 \le r 的突发错误}
$$

### 3.3.5 海明码 ★

海明码的目标是：**检测并纠正单比特错误**。

对于 $k$ 位数据，需要加入 $r$ 位校验位。为了能够纠正 1 位错误，需要满足：

$$
\boxed{2^r \ge k + r + 1}
$$

其中：

- $k$：数据位数
- $r$：校验位数
- $n = k + r$：码字总长度

常见情况：

| 数据位数 $k$ | 所需校验位 $r$ | 总位数 $n$ | 码率 |
|---:|---:|---:|---:|
| 4 | 3 | 7 | 4/7 |
| 8 | 4 | 12 | 8/12 |
| 16 | 5 | 21 | 16/21 |
| 32 | 6 | 38 | 32/38 |

公式来源可以这样理解：

- 一个 $n$ 位码字共有 $2^n$ 种可能模式。
- 有效数据共有 $2^k$ 个合法码字。
- 每个合法码字要覆盖“自己本身 + 任意一位出错”的情况，共 $1+n$ 种模式。
- 因此必须满足：

$$
(1+n) \cdot 2^k \le 2^n
$$

因为 $n = k + r$，化简得到：

$$
2^r \ge k + r + 1
$$

海明码的能力：

- 典型 $(7,4)$ 海明码的最小海明距离为 3。
- 可以检测并纠正 1 位错误。
- 可以检测 2 位错误，但不能可靠纠正 2 位错误。
- 如果强行纠正 2 位错误，可能会被误纠正成另一个错误码字。

成本对比：

| 编码 | 数据位 | 校验位 | 码率 | 能力 |
|---|---:|---:|---:|---|
| 单奇偶校验 | 7 | 1 | 7/8 | 检测奇数个位错误 |
| (7,4) 海明码 | 4 | 3 | 4/7 | 纠正 1 位错误，检测 2 位错误 |

海明码比简单奇偶校验更强，但代价是校验位更多、码率更低。

## 3.4 基本数据链路协议

### 3.4.1 停等协议

停等协议的思想很简单：**发送方一次只发送一帧，收到确认后才发送下一帧**。

基本流程：

1. 发送方发送一帧。
2. 发送方暂停，等待接收方确认。
3. 接收方正确收到后返回 ACK。
4. 发送方收到 ACK 后，再发送下一帧。

为了区分“新帧”和“重传帧”，停等协议通常只需要 1 位序号，在 `0` 和 `1` 之间交替使用。

停等协议的优点是简单，缺点是效率很低。尤其在 RTT 很大的链路上，发送方大部分时间都在等 ACK，信道空着不能用。

### 3.4.2 停等 ARQ

ARQ 的意思是：发现错误或丢失后，靠重传来恢复。停等 ARQ 在停等协议基础上加入三个机制：

- **确认 ACK**：接收方正确收到帧后返回确认。
- **定时器**：发送方发送帧后启动定时器。
- **帧序号**：区分新帧和重传帧，避免重复交付。

帧可能有三种状态：

- **正确到达**：接收方交付给上层，并返回 ACK。
- **出错到达**：接收方检测到错误，丢弃该帧，需要重传。
- **丢失**：接收方根本没有收到，发送方最终靠超时重传。

接收方处理逻辑：

```
收到帧？
├─ 没收到：发送方超时后重传
└─ 收到：
   ├─ 有错误：丢弃，等待重传
   └─ 无错误：交付上层，返回 ACK
```

停等协议的信道利用率常用公式：

$$
\boxed{U = \frac{T_D}{T_D + RTT + T_A}}
$$

其中：

- $T_D$：数据帧发送时延。
- $RTT$：往返时延。
- $T_A$：ACK 发送时延。

如果 ACK 很短，可以近似为：

$$
\boxed{U \approx \frac{T_D}{T_D + RTT}}
$$

例子：链路速率 50 千比特/秒，帧长 1000 比特，RTT = 500 毫秒。

$$
T_D = \frac{1000}{50000} = 0.02s = 20毫秒
$$

$$
U = \frac{20}{20 + 500} \approx 3.8\%
$$

这说明停等协议会严重浪费长时延链路，所以需要滑动窗口协议。

## 3.5 滑动窗口协议 ★★★

### 3.5.1 核心概念

滑动窗口协议的目标是：**允许发送方连续发送多个尚未确认的帧，从而提高信道利用率**。

核心概念：

- **发送窗口 $W_T$**：发送方当前允许发送但还没全部确认的序号范围。
- **接收窗口 $W_R$**：接收方当前愿意接收的序号范围。
- **未确认帧**：已经发送出去，但还没有收到 ACK 的帧。
- **窗口滑动**：收到确认或交付数据后，窗口向前移动。

发送窗口的移动规则：

- 发送新帧时，窗口上界向前扩展。
- 收到 ACK 时，窗口下界向前移动。
- 如果窗口满了，发送方必须暂停，直到收到新的 ACK。

接收窗口的移动规则：

- 接收方收到期望的帧，并能按序交付给上层时，接收窗口向前移动。
- 不同协议对乱序帧的处理不同：GBN 丢弃乱序帧，SR 缓存乱序帧。

### 3.5.2 捎带确认

在全双工通信中，双方都可能同时发送数据。此时 ACK 可以不单独发一帧，而是放在反方向的数据帧里一起带回去，这叫 **捎带确认**。

好处：

- 减少单独 ACK 帧的数量。
- 节省带宽。
- 提高协议效率。

注意：如果接收方迟迟没有反方向数据可发，也不能一直等下去，否则发送方会超时。因此实际协议通常会设置 ACK 延迟上限。

### 3.5.3 三类协议对比

| 协议 | 发送窗口 $W_T$ | 接收窗口 $W_R$ | 特点 |
|---|---:|---:|---|
| 停等 | 1 | 1 | 最简单，效率低 |
| 后退 N 帧 | $1 < W_T \le 2^n - 1$ | 1 | 接收方只按序接收，出错后重传一串 |
| 选择重发 | $1 < W_T \le 2^{n-1}$ | $1 < W_R \le 2^{n-1}$ | 只重传出错帧，效率高但复杂 |

其中 $n$ 是序号字段的比特数。

## 3.6 后退 N 帧 ARQ ★★★

后退 N 帧协议简称 GBN。它允许发送方连续发送多个帧，但接收方只接受按序到达的帧。

### 3.6.1 发送方行为

发送方维护一个发送窗口，最多允许 $W_T$ 个未确认帧在路上。

规则：

1. 只要发送窗口未满，就可以继续发送新帧。
2. 收到 ACK 后，发送窗口向前滑动。
3. ACK 是累计确认。例如 ACK 5 表示 5 之前的帧都已经正确收到，接收方下一帧想要 5。
4. 如果最早未确认帧超时，发送方从该帧开始，把它以及后面所有未确认帧全部重传。

这就是“后退 N 帧”的含义：出错后不是只重传一个帧，而是回退到出错处，把后面一串都重传。

### 3.6.2 接收方行为

GBN 的接收窗口大小为 1。

接收方规则：

1. 只接受当前期望序号的帧。
2. 如果帧序号正确且无错，就交付给上层，并返回下一个期望序号的 ACK。
3. 如果收到乱序帧，即使该帧本身没有错误，也直接丢弃。
4. 对乱序帧，接收方通常重复发送最近一次的正确 ACK。

例子：接收方期望收到 3，但收到了 4。

```
期望：3
收到：4
处理：丢弃 4，继续请求 3
```

### 3.6.3 效率分析

设：

- $T_f$：发送一帧所需时间。
- $T_p$：单向传播时延。
- $W$：发送窗口大小。

如果窗口足够大，使发送方在第一个 ACK 回来前一直有帧可发，则信道可以接近满载：

$$
W \cdot T_f \ge T_f + 2T_p
$$

窗口不够大时，近似利用率为：

$$
\boxed{U = \frac{W \cdot T_f}{T_f + 2T_p}}
$$

最大利用率不超过 1：

$$
\boxed{U = \min\left(1, \frac{W \cdot T_f}{T_f + 2T_p}\right)}
$$

有错误时，GBN 效率会明显下降，因为一个帧出错可能导致后面多个已经正确到达的帧也被重传。

### 3.6.4 常见差错情况

**情况 1：数据帧损坏**

接收方检测到某帧错误，会丢弃该帧。后续乱序帧也会被丢弃。发送方超时后，从出错帧开始重传。

**情况 2：数据帧丢失**

如果帧 3 丢失，接收方之后收到帧 4、5、6，也会因为不是期望序号而丢弃，并重复请求帧 3。发送方超时后从帧 3 开始重传。

**情况 3：ACK 丢失**

由于 GBN 使用累计确认，单个 ACK 丢失不一定有问题。比如 ACK 3 丢了，但 ACK 5 到了，那么发送方可以知道 5 之前的帧都已经被确认。

### 3.6.5 GBN 窗口大小为什么是 $2^n - 1$

如果序号字段有 $n$ 位，则序号空间大小是 $2^n$。

GBN 要求：

$$
\boxed{W_T \le 2^n - 1}
$$

原因是避免新帧和旧帧重传发生歧义。

例子：若 $n=3$，序号为 0 到 7。如果窗口大小允许为 8，发送方可能一次把 0 到 7 全发出去。等序号绕回后，接收方看到新的 0，可能分不清它是“新一轮的 0”，还是“上一轮的 0 的重传”。

所以必须至少留出一个序号不用，让新旧窗口不会完全重叠。

## 3.7 选择重发 ARQ ★★★

选择重发简称 SR。它比 GBN 更精细：**哪个帧错了或丢了，就只重传哪个帧**。

### 3.7.1 发送方行为

SR 发送方规则：

1. 发送方可以连续发送窗口内的多个帧。
2. 每个未确认帧都有自己的定时器。
3. 某个帧超时，只重传这个帧。
4. 收到某个帧的 ACK，只确认该帧。
5. 当窗口左边连续的帧都被确认后，发送窗口向前滑动。

SR 不能像 GBN 那样完全依赖累计确认，因为它允许乱序接收。

### 3.7.2 接收方行为

SR 的接收窗口大于 1，可以接收乱序帧。

接收方规则：

1. 收到窗口内的正确帧，就先缓存起来，并返回该帧的 ACK。
2. 如果缺少前面的帧，后面的帧不能立刻交付给上层，只能暂存在缓冲区。
3. 等缺失帧到达后，接收方把连续的一段按序交付给上层。
4. 收到窗口外的帧，通常丢弃；必要时重复 ACK。

例子：接收方期望 3，但先收到了 4 和 5。

```
帧 4：缓存
帧 5：缓存
帧 3：到达后，3、4、5 一起按序交付
```

### 3.7.3 NAK 的作用

NAK 是否定确认，意思是“我缺少某个帧，请你重传”。

在 SR 中，NAK 可以提高效率：

- 没有 NAK：发送方要等定时器超时才重传。
- 有 NAK：接收方一发现缺口，就可以主动请求重传。

但 NAK 不是正确性的必要条件。即使 NAK 丢失，发送方的定时器最终也会超时并触发重传。

### 3.7.4 SR 的缺点

SR 的效率通常高于 GBN，但实现更复杂：

- 每个帧都要单独计时。
- 每个帧都要单独确认。
- 接收方需要缓存乱序帧。
- 发送方和接收方都要维护更复杂的窗口状态。

无错误时，SR 和 GBN 的效率差不多；有错误时，SR 往往更好，因为它避免重传已经正确收到的帧。

### 3.7.5 时序图做题方法

时序图通常用来考 ARQ 协议过程。

读图时注意：

- 时间通常从上往下流动。
- 从左到右的箭头表示数据帧。
- 从右到左的箭头表示 ACK 或 NAK。
- 箭头上会标帧号，例如 0、1、2。
- 叉号或断线通常表示帧丢失或损坏。

做题时先判断协议：

- 停等：一发一等。
- GBN：乱序帧丢弃，出错后重传一串。
- SR：乱序帧缓存，只重传缺失帧。

### 3.7.6 GBN 和 SR 怎么选

| 场景 | 更适合的协议 | 原因 |
|---|---|---|
| 错误率较高 | SR | 避免大量无谓重传 |
| 接收方实现要简单 | GBN | 接收窗口为 1，不缓存乱序帧 |
| 带宽高、传播时延长 | SR | 更能充分利用链路 |
| 缓冲区很有限 | GBN | 接收方压力小 |
| 希望节省重传带宽 | SR | 只重传真正缺失的帧 |

## 3.8 帧序号字段设计 ★★★

这一节常考计算题：给定带宽、传播时延、帧长，问序号字段至少需要多少位。

核心步骤：

### 第一步：计算一帧发送时间

$$
\boxed{T_f = \frac{L}{R}}
$$

其中：

- $L$：帧长，单位 比特。
- $R$：链路速率，单位 比特/秒。

### 第二步：计算 RTT 内能发送多少帧

$$
\frac{RTT}{T_f}
$$

为了让发送方在第一个 ACK 回来之前不空等，所需窗口大约为：

$$
\boxed{W_{所需} = \left\lceil 1 + \frac{RTT}{T_f} \right\rceil}
$$

有些题目会用带宽时延积：

$$
\boxed{\frac{R \cdot RTT}{L}}
$$

它表示一个 RTT 内链路上大约能容纳多少帧。

### 第三步：根据协议套序号约束

GBN：

$$
\boxed{2^n - 1 \ge W_{所需}}
$$

所以：

$$
\boxed{n = \left\lceil \log_2(W_{所需}+1) \right\rceil}
$$

SR：

$$
\boxed{2^{n-1} \ge W_{所需}}
$$

所以：

$$
\boxed{n = \left\lceil \log_2 W_{所需} \right\rceil + 1}
$$

### 例题 1

链路速率 1.6 兆比特/秒，RTT = 45 毫秒，帧长 1 KB，求最大吞吐量时所需序号位数。

一帧发送时间：

$$
T_f = \frac{1 \times 1024 \times 8}{1.6 \times 10^6} = 5.12毫秒
$$

所需窗口：

$$
W_{所需} = \left\lceil 1 + \frac{45}{5.12} \right\rceil = \lceil 9.79 \rceil = 10
$$

如果使用 GBN：

$$
2^n - 1 \ge 10 \Rightarrow 2^n \ge 11 \Rightarrow n = 4
$$

如果使用 SR：

$$
2^{n-1} \ge 10 \Rightarrow n = 5
$$

### 例题 2

4 兆比特/秒 点到点链路，距离 5000 千米，传播速度 $3 \times 10^8$m/s，帧长 1 KB，求 SR 至少需要多少序号位。

单向传播时延：

$$
T_p = \frac{5000 \times 10^3}{3 \times 10^8} \approx 16.67毫秒
$$

往返时延：

$$
RTT \approx 33.33毫秒
$$

一帧发送时间：

$$
T_f = \frac{1024 \times 8}{4 \times 10^6} = 2.048毫秒
$$

所需窗口：

$$
W_{所需} = \left\lceil 1 + \frac{33.33}{2.048} \right\rceil = \lceil 17.28 \rceil = 18
$$

SR 约束：

$$
2^{n-1} \ge 18
$$

所以：

$$
n = 6
$$

## 3.9 点对点协议 PPP ★

PPP 用于点到点链路，例如早期拨号接入、某些串行链路等。它主要解决三件事：成帧、链路控制、网络层协议协商。

### 3.9.1 PPP 的三个组成部分

1. **成帧方法**：把网络层数据报封装成 PPP 帧。
2. **链路控制协议 LCP**：建立、配置、测试、维护和终止链路。
3. **网络控制协议 NCP**：协商网络层协议参数，例如 IP 相关参数。

PPP 本身提供的是不可靠传输：它可以检错，但不负责纠错、流量控制和可靠重传。

### 3.9.2 PPP 帧格式

```
标志 | 地址 | 控制 | 协议 | 信息字段 | FCS | 标志
```

各字段含义：

| 字段 | 默认长度 | 说明 |
|---|---:|---|
| 标志 | 1 B | 固定为 `0x7E`，表示帧边界 |
| 地址 | 1 B | 固定为 `0xFF` |
| 控制 | 1 B | 固定为 `0x03` |
| 协议 | 2 B | 指出信息字段承载的协议，可协商为 1 B |
| 信息字段 | 0 到 1500 B | 默认最大 1500 B，可协商 |
| FCS | 2 B | 帧校验序列，可协商为 4 B |
| 标志 | 1 B | 固定为 `0x7E` |

PPP 使用字节填充实现透明传输：如果数据中出现 `0x7E`，就要用转义方式避免被误认为帧边界。

### 3.9.3 PPP 帧长度计算

默认长度：

$$
1 + 1 + 1 + 2 + 1500 + 2 + 1 = \boxed{1508B}
$$

最小长度：

$$
1 + 1 + 1 + 1 + 0 + 2 + 1 = \boxed{7B}
$$

一种常见最大情况：

$$
1 + 1 + 1 + 2 + 1500 + 4 + 1 = \boxed{1510B}
$$

考试注意：题目如果明确说协议字段、信息字段或 FCS 已经协商过，就按题目给出的字段长度算。

## 3.10 高级数据链路控制 HDLC

HDLC 是一种面向比特的数据链路协议，使用比特填充实现透明传输。

帧边界标志为：

```
01111110
```

为了避免数据中出现同样的比特串，发送方在数据中每遇到连续 5 个 `1`，就自动插入一个 `0`。接收方收到后，看到连续 5 个 `1` 后面的 `0`，就把这个 `0` 删除。

### 3.10.1 帧类型

| 类型 | 中文名 | 作用 |
|---|---|---|
| I 帧 | 信息帧 | 携带用户数据，也携带发送序号和确认序号 |
| S 帧 | 监督帧 | 用于确认、流量控制、差错控制 |
| U 帧 | 无编号帧 | 用于链路管理，例如建立连接、断开连接、设置模式 |

常见监督帧功能：

- **RR**：接收方准备好，可以继续接收。
- **RNR**：接收方暂时未准备好。
- **REJ**：拒绝，从某帧开始重传。
- **SREJ**：选择拒绝，只请求重传某一帧。

### 3.10.2 工作模式

| 模式 | 中文名 | 特点 |
|---|---|---|
| NRM | 正常响应模式 | 主站控制从站，从站被轮询后才能发送 |
| ARM | 异步响应模式 | 从站可以不等轮询主动发送 |
| ABM | 异步平衡模式 | 双方地位平等，常用于全双工点到点链路 |

## 3.11 练习题

### 填空题

1. ARQ 协议的三个核心机制是：______、______、______。
   答案：确认、定时器、帧序号。

2. 后退 N 帧协议中，接收窗口大小为 ______。
   答案：1。

3. PPP 帧以标志字节 ______ 开始和结束。
   答案：`0x7E`。

4. HDLC 使用 ______ 填充实现透明传输。
   答案：比特填充。

5. 要纠正 1 位错误，最小海明距离至少为 ______。
   答案：3。

6. 选择重发协议中，若序号字段为 $n$ 位，最大发送窗口为 ______。
   答案：$2^{n-1}$。

7. CRC 是一种 ______ 码。
   答案：检错码。

8. PPP 默认协议信字段长度为 ______ 字节，可协商为 ______ 字节。
   答案：2，1。

9. 四种成帧方法是：______、______、______、______。
   答案：字节计数法、字节填充法、比特填充法、物理层编码违例法。

10. 比特填充方法中的典型标志比特串是 ______。
    答案：`01111110`。

### 判断题

1. 停等 ARQ 需要 2 位序号。
   答案：错。停等只需要 1 位序号，在 0 和 1 间交替。

2. GBN 接收方会缓存乱序帧。
   答案：错。GBN 丢弃乱序帧，SR 才缓存乱序帧。

3. CRC 可以纠正单比特错误。
   答案：错。CRC 只能检错，不能纠错。

4. 3 位序号的 GBN 最大发送窗口是 8。
   答案：错。最大为 $2^3 - 1 = 7$。

5. SR 中的 NAK 是正确性必须条件。
   答案：错。NAK 是性能优化；没有 NAK 也可以靠超时重传。

6. PPP 使用字节填充，HDLC 使用比特填充。
   答案：对。

### 选择题

1. 哪个协议要求接收方缓存乱序帧？

   a. 停等  
   b. 后退 N 帧  
   c. 选择重发  
   d. 都需要

   答案：c。

2. GBN 使用 3 位序号时，最大发送窗口为：

   a. 7  
   b. 8  
   c. 4  
   d. 3

   答案：a。

3. 若生成多项式为 $G(x)=x^4+x+1$，CRC 余数有几位？

   a. 1 位  
   b. 2 位  
   c. 3 位  
   d. 4 位

   答案：d。余数位数等于生成多项式次数。

4. PPP 帧中的地址字段通常固定为：

   a. `0x00`  
   b. `0xFF`  
   c. `0x7E`  
   d. `0x03`

   答案：b。

### 计算题

**题 1：停等利用率**

链路速率 1 兆比特/秒，RTT = 20 毫秒，帧长 1000 B，忽略 ACK 发送时间，求停等利用率。

$$
T_D = \frac{1000 \times 8}{10^6} = 8毫秒
$$

$$
U = \frac{8}{8+20} = 28.6\%
$$

**题 2：GBN 序号位数**

若 GBN 需要发送窗口至少为 20，求序号字段至少多少位。

GBN 满足：

$$
2^n - 1 \ge 20
$$

$$
2^n \ge 21
$$

所以：

$$
n = 5
$$

**题 3：SR 序号位数**

若 SR 需要发送窗口至少为 20，求序号字段至少多少位。

SR 满足：

$$
2^{n-1} \ge 20
$$

$$
n-1 \ge \lceil \log_2 20 \rceil = 5
$$

所以：

$$
n = 6
$$

**题 4：PPP 默认帧长**

PPP 默认信息字段为 1500 B，协议字段 2 B，FCS 2 B，求默认帧长。

$$
1 + 1 + 1 + 2 + 1500 + 2 + 1 = 1508B
$$
